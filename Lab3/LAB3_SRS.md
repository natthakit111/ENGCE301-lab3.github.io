# SRS – ENGCE301 Lab Submission Portal

## 1. Introduction

### 1.1 Purpose (วัตถุประสงค์)
เอกสาร SRS ฉบับนี้กำหนดความต้องการของระบบ **ENGCE301 Lab Submission Portal**  
เพื่อใช้เป็นกรณีศึกษาในวิชา **ENGCE301 – Software Design & Development** โดยมีเป้าหมาย:

- ให้นักศึกษาฝึกอ่าน/วิเคราะห์ SRS
- แปลง SRS → User Stories → Use Cases
- นำ Use Cases ไปออกแบบและพัฒนา **หน้าเว็บจริง (HTML/CSS)** ใน LAB Week 3

### 1.2 Scope (ขอบเขต)
ระบบ Lab Submission Portal จะช่วย:

- นักศึกษา:  
  - ดูรายการ LAB ทั้งหมดของรายวิชา  
  - ดูรายละเอียด LAB และกำหนดส่ง  
  - ส่งไฟล์งาน LAB ผ่านเว็บ และดูสถานะ / คะแนน
- ผู้สอน / TA:  
  - สร้างและจัดการ LAB  
  - ดูรายการส่งงานของนักศึกษา  
  - ให้คะแนนและ Feedback

สcope ของ SRS นี้เน้นฟังก์ชันหลักสำหรับวิชานี้ **ยังไม่ครอบคลุม** ระบบ login จริง, การ integrate กับระบบมหาวิทยาลัย ฯลฯ (ถือเป็น out-of-scope ในวิชานี้)

### 1.3 Actors & Roles

- **Student**  
  - ผู้ส่งงาน LAB, ดูรายละเอียด LAB, ดูคะแนน/สถานะ
- **Instructor** (ผู้สอน)  
  - ผู้สร้าง LAB, แก้ไข LAB, ให้คะแนน/Feedback
- **Teaching Assistant (TA)** (อาจมี/ไม่มีก็ได้)  
  - ช่วย Instructor ดู submission และให้คะแนนในบางส่วน (optional)

### 1.4 Definitions, Acronyms
- **Lab** – งานทดลอง/การบ้านประจำสัปดาห์
- **Submission** – การส่งงาน LAB หนึ่งครั้งของนักศึกษา 1 คน
- **Deadline** – วัน–เวลาหมดเขตส่งงานของ LAB
- **Late Submission** – การส่งงานหลัง deadline ตาม policy ที่ผู้สอนกำหนด

## 2. Overall Description

### 2.1 System Overview
ระบบเป็นเว็บแอปพลิเคชันที่แสดงรายการ LAB ของวิชา ENGCE301,  
อนุญาตให้นักศึกษาอัปโหลดงาน, ดูสถานะการส่ง, และให้อาจารย์ตรวจ–ให้คะแนนผ่านระบบเดียวกัน

สำหรับวิชานี้ นักศึกษาจะสร้าง **หน้าเว็บต้นแบบ (Prototype UI)**  
เช่น Landing Page, My Labs Page, About Page โดยใช้ HTML/CSS เท่านั้น (ยังไม่ต้องมี backend จริง)

### 2.2 Assumptions
- ผู้ใช้ใช้เว็บเบราว์เซอร์สมัยใหม่ (Chrome, Edge, Firefox, ฯลฯ)
- ระบบจริงในอนาคตอาจต้องเชื่อมต่อกับระบบบัญชีมหาวิทยาลัย (Single Sign-On)  
  แต่ในรายวิชานี้ถือว่า “mock login / assume already logged in”

## 3. Functional Requirements (FR)

> *หมายเหตุ*: แต่ละ FR สามารถ map ไปเป็น Use Case ได้ 1 ตัว หรือมากกว่า

### FR-1 View Lab List
**Description**  
ระบบต้องแสดงรายการ LAB ทั้งหมดของรายวิชาให้กับนักศึกษา

**Details**
- แสดง: รหัส/ชื่อ LAB, คำอธิบายสั้น ๆ, วัน–เวลาหมดเขตส่ง (deadline), สถานะ (open/closed)
- สามารถเรียงตาม “ลำดับสัปดาห์” หรือ “วันหมดเขต” ได้ (อย่างน้อย 1 แบบ)

---

### FR-2 View Lab Details
**Description**  
นักศึกษาสามารถเปิดดูรายละเอียดของ LAB แต่ละตัวได้

**Details**
- แสดง: ชื่อ LAB, จุดประสงค์การเรียนรู้ (Objectives), รายละเอียดโจทย์, ไฟล์แนบ (เช่น PDF), deadline
- แสดงสถานะการส่งงานของนักศึกษาคนนั้น (ยังไม่ส่ง / ส่งแล้ว / ส่งช้า / ได้คะแนนแล้ว ฯลฯ)

---

### FR-3 Submit Lab
**Description**  
นักศึกษาสามารถอัปโหลดไฟล์ส่ง LAB ผ่านระบบได้

**Details**
- รองรับไฟล์ `.pdf`, `.docx`, `.zip` อย่างน้อย 1 ชนิด
- บันทึกเวลาในการส่งงาน (timestamp)
- หากส่งก่อน deadline ถือว่า “ส่งตรงเวลา”
- ถ้าส่งหลายครั้งก่อน deadline ให้ระบบนับ submission ล่าสุดเป็นตัวหลัก

---

### FR-4 Handle Late Submission
**Description**  
ระบบต้องรองรับ / ปฏิเสธการส่งงานหลัง deadline ตาม Policy ที่ผู้สอนกำหนด

**Details (แนะนำ)**  
ตัวอย่าง policy (เลือกใช้ 1 อย่าง หรือผสม):

- แบบที่ 1: ไม่รับส่งงานหลัง deadline
- แบบที่ 2: ให้ส่งได้แต่ติดสถานะ "Late"
- แบบที่ 3: ให้ส่งได้แต่หักคะแนนอัตโนมัติ

*(ในวิชานี้ อาจารย์สามารถกำหนด policy ให้ นศ. แปลงเป็น Use Case และ UI ได้)*

---

### FR-5 Manage Labs (Instructor)
**Description**  
ผู้สอนสามารถสร้าง/แก้ไข LAB ได้

**Details**
- สร้าง LAB ใหม่ได้ โดยระบุ: ชื่อ, คำอธิบาย, deadline, ไฟล์แนบ (ถ้ามี)
- แก้ไขข้อมูล LAB ที่ยังไม่หมดเขตได้
- ปิด LAB (ไม่รับส่งแล้ว) เมื่อหมดเขตหรือเมื่อผู้สอนต้องการ

---

### FR-6 View Submissions (Instructor / TA)
**Description**  
ผู้สอน/TA สามารถดูรายการส่งงานของนักศึกษาแต่ละ LAB ได้

**Details**
- แสดงรายชื่อนักศึกษาทั้งหมด และสถานะการส่ง/ไม่ส่ง
- สามารถดาวน์โหลดไฟล์ที่ส่งได้

---

### FR-7 Grade & Feedback
**Description**  
ผู้สอน/TA สามารถให้คะแนนและ Feedback แต่ละ submission ได้

**Details**
- ระบุคะแนน (เช่น 0–100) และข้อความ Feedback
- บันทึกวันที่/เวลาที่ให้คะแนน
- นักศึกษาสามารถเห็นคะแนนและ Feedback ของตนเองในหน้าสถานะ

---

### FR-8 View My Lab Status (Student)
**Description**  
นักศึกษาสามารถดูภาพรวมสถานะ LAB ทุกตัวของตนเองได้บนหน้า “My Labs”

**Details**
- แสดงรายการ LAB พร้อมสถานะ: ยังไม่ส่ง / ส่งแล้ว / ส่งช้า / ตรวจแล้ว
- แสดงคะแนนถ้ามี และ link ไปดูรายละเอียด LAB แต่ละตัว

---

## 4. Non-Functional Requirements (NFR)

### NFR-1 Usability
- UI ต้องอ่านง่าย เหมาะกับการใช้งานบนจอ Desktop/Laptop สำหรับการเรียน
- ใช้ layout ที่สม่ำเสมอ เช่น header, navigation bar, content area

### NFR-2 Performance
- หน้าแสดงรายการ LAB ควรโหลดภายใน ~3 วินาที (สำหรับ prototype ให้เน้นไม่ใส่ของหนักเกิน เช่น รูปภาพใหญ่ ๆ)

### NFR-3 Security (ระดับวิชาเรียน)
- ในวิชานี้ ถือว่ายังไม่ต้องทำระบบ login จริง แต่ออกแบบ UI ให้มีที่สำหรับแสดงชื่อผู้ใช้ / บทบาท  
  เพื่อเตรียมต่อยอดไปสู่ระบบจริงได้ในอนาคต

### NFR-4 Maintainability
- HTML/CSS ต้องถูกแยกไฟล์ชัดเจน (`index.html`, `styles.css` ฯลฯ)
- ใช้ class / id ที่สื่อความหมาย เพื่อให้อ่าน/แก้ไขโค้ดง่าย
